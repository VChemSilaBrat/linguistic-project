## 2. Root Cause: Span-F1 vs Token-F1 Gap

Token-F1 ~0.83 означает, что модель неплохо различает "математические" и "обычные" токены. Но span-F1 ~0.19 — модель не может правильно определить **границы** спанов (где B-MATH, а где продолжить I-MATH, и где остановиться).

### Почему? Несовпадение характеристик спанов между данными:

| Dataset         | Avg span len | Single-token spans | Spans >= 10 tokens | Math token ratio |
|-----------------|-------------|-------------------|-------------------|-----------------|
| train_clean (synthetic, 185 sent) | 10.0 | 0% | 51% | 0.698 |
| train_noisy (Mathematicon, 1261 sent) | 2.8 | 52% | 3% | 0.346 |
| **test (Gosha, 50 sent)** | **6.8** | **25%** | **25%** | **0.516** |

Ключевые наблюдения:
- **Synthetic (train_clean)** — спаны слишком длинные и "чистые". Модель учится генерировать длинные непрерывные спаны, но в тесте спаны короче и разнообразнее.
- **Mathematicon (train_noisy)** — спаны слишком короткие и шумные. 52% — одиночные токены. Не учит модель находить границы длинных спанов.
- **Gosha (test)** — посередине: средняя длина 6.8, разброс от 1 до 15+.

## 3. Проблемы с Mathematicon (train_noisy)

Конвертация из WebAnno TSV использует gap-merge heuristic (gap<=2, без merge через пунктуацию .!?;). Проблемы:

### 3.1. Ложные спаны (137+ случаев)
Слова, размеченные как B-MATH, но не являющиеся математикой:
- "понятности", "случаю", "предметов", "свойствами", "этом"
- "этом каждый предмет" размечено как один спан

### 3.2. Пунктуация внутри спанов
446 знаков пунктуации (9.8% всех I-MATH) внутри спанов: запятые, точки. Это результат gap-merge через "," (запятая не входит в SENT_BREAK_PUNCT = .!?;).

Пример: "свойствами альфа один , альфа два , альфа эн" — запятые внутри спана.

### 3.3. Слишком агрессивный merge
gap<=2 сливает явно разные выражения, если между ними стоит только 1-2 обычных слова.

### Возможные исправления Mathematicon конвертации:
1. **Добавить запятую в SENT_BREAK_PUNCT** (или по крайней мере не мерджить через запятую + не-мат слово)
2. **Уменьшить MAX_GAP до 1** или сделать gap=0 (merge только непрерывных math-токенов)
3. **Фильтр по markers.yaml** — использовать роли маркеров для валидации: если одиночный B-MATH токен не входит ни в одну категорию markers.yaml, удалить его из разметки
4. **Фильтровать предложения** с очевидно шумной разметкой (например, где >50% токенов — MATH, но нет ни одного оператора/переменной по markers.yaml)

## 4. Идея: Аугментация через s2l

data/s2l/russian_sentences_train.csv содержит **71,170 пар** (pronunciation, formula):
```
пять икс плюс три равно минус семь икс плюс восемь | 5x+3=-7x+8
```

Это чистые математические выражения, произнесённые по-русски. Сейчас не используются.

### Как использовать:
1. **Каждая строка s2l = один полный спан** (pronunciation — это целиком B-MATH I-MATH I-MATH...)
2. **Обернуть в несущие фразы** из лекционного корпуса, например:
   - "Рассмотрим выражение [MATH]пять икс плюс три равно минус семь икс плюс восемь[/MATH]."
   - "Получаем, что [MATH]...[/MATH], что и требовалось доказать."
   - "Если [MATH]...[/MATH], то решение единственно."
3. Шаблоны несущих фраз можно взять из Gosha/synthetic данных (контексты вокруг спанов)
4. **Результат:** тысячи дополнительных тренировочных примеров с разнообразной длиной спанов и чистой разметкой

### Преимущества:
- Огромный объём данных (71k vs 185 synthetic)
- Чистая разметка (каждый pronunciation — гарантированно math)
- Разнообразие: от простых (2-3 токена) до длинных выражений
- Модель увидит больше boundary-паттернов (переходы O->B-MATH и I-MATH->O)

### Риски:
- Carrier phrases будут шаблонными — модель может переобучиться на шаблоны вместо мат-контента
- s2l — чисто формульная речь без лекционного контекста (нет "ну вот", "значит", "смотрите")
- Нужно оценить: сколько уникальных carrier-фраз достаточно? 50? 200?

## 5. Идея: Использовать markers.yaml (от Димы) для улучшения

data/dima/markers.yaml содержит ~1650 строк с >100 паттернами, распределёнными по ролям:
- VARIABLE: "икс", "альфа", "дельта", латинские буквы
- OPERATOR: "плюс", "минус", "интеграл", "производная", "синус"
- RELATION: "равно", "больше", "стремится к", "принадлежит"
- STRUCTURE: "в степени", "корень из", "дробь", "модуль"
- QUANTIFIER: "для любого", "существует"
- DISCOURSE_START: "пусть", "рассмотрим", "получаем"
- BOUNDARY: "где", "то есть", "если"
- NEGATIVE_CONTEXT: "обратите внимание", "на практике"

Каждый паттерн имеет position scores (start/inside/end) и confidence.

### Применения:
1. **Улучшить Mathematicon конвертацию** — фильтровать шумные спаны через markers
2. **CRF features** (E5/E6) — добавить marker_role и marker_position как фичи
3. **BERT: marker-guided data selection** — из Mathematicon оставить только предложения, где размеченные спаны содержат хотя бы один high-confidence маркер
