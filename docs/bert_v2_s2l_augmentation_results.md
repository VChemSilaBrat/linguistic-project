# BERT v2: Эксперименты с s2l аугментацией

**Дата:** 2026-06-16  
**Статус:** Завершено. Аугментация через s2l ухудшила результаты.

---

## 1. Цель

Проверить гипотезу: добавление данных из s2l (71k пар «произношение ↔ формула») улучшит span-F1 за счёт разнообразия длин спанов и увеличения объёма тренировочных данных.

## 2. Метод аугментации

Использовалась **template injection**: из 185 синтетических предложений извлекались шаблоны (контекст вокруг математических спанов), затем спаны заменялись на произношения формул из s2l.

- Источник: `data/s2l/russian_sentences_train.csv` (71,108 пар)
- Фильтр: s2l произношения длиной 3–20 токенов (для разнообразия длин)
- Объём аугментации: 2000 сгенерированных предложений (`train_s2l_aug`)
- Разметка: каждое вставленное произношение = один сплошной спан (B-MATH + I-MATH...)

## 3. Эксперименты

### BERT-2: Clean + s2l аугментация
- **Данные:** train_clean (185 synthetic) + train_s2l_aug (2000 s2l)
- **Конфигурация:** LR 2e-5, 10 эпох, early stopping patience=3
- **3 сида:** 42, 123, 456

### BERT-3: Curriculum + s2l аугментация
- **Stage 1:** train_noisy (Mathematicon), LR 3e-5, 3 эпохи, label_smoothing=0.1
- **Stage 2:** train_clean + train_s2l_aug, fresh optimizer, LR 1.5e-5, 15 эпох, patience=3
- **3 сида:** 42, 123, 456

## 5. Анализ: почему аугментация ухудшила результаты

### 5.1. Закономерность: больше данных → хуже результат

Чёткая отрицательная корреляция между объёмом тренировочных данных и качеством:
- BERT-0 (185 предложений) → 0.1932 — лучший
- BERT-2 (2185 предложений) → 0.1634 — хуже
- BERT-1/BERT-3 (+Mathematicon) → ещё хуже

### 5.2. Вероятные причины

**Разбавление сигнала (signal dilution).** 185 синтетических предложений, видимо, хорошо соответствуют распределению тестовых данных Gosha. Добавление 2000 s2l-предложений «разбавляет» этот сигнал — модель тратит capacity на запоминание паттернов из s2l, которые плохо переносятся на реальные лекции.

**Шаблонность carrier-фраз.** Template injection порождает предложения с одинаковыми контекстами вокруг спанов (всего ~185 уникальных шаблонов). Модель переобучается на эти контексты как на маркеры границ, а в тесте видит совершенно другие контексты.

**Доменное несоответствие.** s2l — это чтение формул вслух («пять икс плюс три равно...»), а Gosha — это лекционная речь с дискурсивными маркерами, оговорками, сложными описаниями. Разные языковые регистры.

**Curriculum усиливает проблему.** Mathematicon добавляет шум (52% одиночных спанов), а s2l не компенсирует его — наоборот, вносит дополнительное доменное смещение.

### 5.3. Что это значит

Проблема span boundary detection для данной задачи **не решается наращиванием данных** из доступных источников. Ключевой bottleneck — это gap между **синтетическими/формульными данными** и **реальной лекционной речью**. Для существенного улучшения нужна либо ручная разметка реальных лекций (дорого), либо принципиально другой подход к аугментации (генерация через LLM в стиле реальных лекций).

## 6. Выводы для защиты

1. **Baseline BERT-0 (185 clean sentences) — лучшая модель.** Минимализм побеждает: малый, но качественный и релевантный датасет даёт лучший результат.
2. **Curriculum learning не помогает** при сильном доменном несоответствии noisy данных — Mathematicon слишком шумный и слишком отличается по характеристикам спанов.
3. **Наивная аугментация из s2l не работает** — template injection и carrier phrases не создают достаточно реалистичного лекционного контекста.
4. **Token-F1 >> Span-F1** для всех моделей — модели хорошо распознают математические токены, но плохо определяют границы спанов. Это фундаментальная проблема задачи при ограниченных данных.
5. **Рекомендация:** для прикладного использования лучше всего работает token-level классификация с постпроцессингом (например, объединение соседних B/I-MATH токенов через правила).
