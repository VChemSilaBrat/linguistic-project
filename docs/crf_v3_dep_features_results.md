# CRF v3: Эксперименты с dependency-фичами Димы

**Дата:** 2026-06-18  
**Статус:** Завершено. CRF-10 (dep + dima_bio) = **0.3612** — лучшая модель проекта, обгоняет BERT-4 (0.3287).

---

## 1. Цель

Интегрировать фичи из Диминого rule-based детектора (`formula_bio.py`) в CRF и проверить, дают ли синтаксические и детекторные признаки дополнительный прирост поверх LLM-данных.

## 2. Что даёт `formula_bio.py`

Димин скрипт — полноценный rule-based детектор математических спанов на основе Stanza + markers.yaml. Пайплайн:

1. **Stanza** (tokenize, POS, lemma, depparse) → для каждого токена: UPOS, лемма, dependency relation (`deprel`), голова в дереве зависимостей
2. **Marker matching** — поиск паттернов из markers.yaml (phrase/exact/lemma/regex) с ролями RELATION, OPERATOR, VARIABLE, STRUCTURE и т.д.
3. **Dependency span expansion** — от «якорных» маркеров расширение спанов по дереву зависимостей (subtree, relation expansion, anchored expansion)
4. **Post-processing** — trim по boundary маркерам, merge соседних спанов, split по gaps между якорями

Из этого пайплайна мы извлекли две фичи для каждого токена:

- **`dep_label`** — dependency relation из Stanza (nsubj, obj, obl, conj, case, nmod, root и т.д.). Отражает синтаксическую роль токена в предложении.
- **`dima_bio`** — BIO-предсказание rule-based детектора (B-MATH / I-MATH / O). По сути «мнение второй модели», основанное на совершенно другом подходе (деревья зависимостей + словарь маркеров).

## 3. Извлечение фичей

**Скрипт:** `scripts/features/extract_dima_features.py`

Для каждого предложения в bio_merged.tsv:
1. Реконструкция текста: `" ".join(tokens)`
2. Stanza parsing + Dima's detect pipeline
3. Alignment выхода Stanza с нашей токенизацией (1:1 при совпадении длин, character-level alignment при расхождении)

**Результаты извлечения:**
- 2035 предложений обработано
- 128 предложений — чистое 1:1 выравнивание
- 1907 предложений — расхождение длин (Stanza разбивает/объединяет токены иначе)
- 5236 из 45422 токенов (11.5%) — UNK fallback
- **На используемых сплитах (clean, llm, val, test) — 100% покрытие**, все 23945 токенов получили фичи

Расхождения в основном на train_noisy (Mathematicon), который мы не используем в лучших конфигурациях.

## 6. Анализ

### 6.1. Вклад каждой фичи

| Фича | Изолированный эффект (Δ vs CRF-2 base) |
|------|----------------------------------------|
| dep_label | +0.0124 (0.3356 vs 0.3232) |
| dima_bio | +0.0270 (0.3502 vs 0.3232) |
| dep + dima_bio | +0.0380 (0.3612 vs 0.3232) |

`dima_bio` — более сильная фича (+2.7 пп), чем `dep_label` (+1.2 пп). Их комбинация даёт суперлинейный эффект (+3.8 пп > 1.2 + 2.7 = 3.9 пп — практически аддитивный).

### 6.2. POS вредит при наличии dima_bio

| Фичи | Без POS | С POS | Δ |
|------|---------|-------|---|
| dep | 0.3356 (CRF-6) | 0.3510 (CRF-7) | **+0.0154** |
| dima_bio | 0.3502 (CRF-8) | 0.3377 (CRF-9) | **−0.0125** |
| dep+dima_bio | 0.3612 (CRF-10) | 0.3475 (CRF-11) | **−0.0137** |

POS помогает только в комбинации с dep_label. Когда dima_bio присутствует, POS вредит. Объяснение: Dima's детектор уже использует POS внутренне (Stanza POS → marker matching → span expansion). Добавление POS-фичи в CRF создаёт избыточность, увеличивая пространство признаков без нового сигнала, что ведёт к overfitting на 694 тренировочных предложениях.

### 6.3. Почему dima_bio работает как фича

`dima_bio` — по сути «мнение эксперта», закодированное как правила. Детектор Димы использует:
- Лексический словарь (markers.yaml, 641 слово с ролями)
- Синтаксическую структуру (dependency tree expansion)
- Пост-обработку (trim, merge, split по boundary маркерам)

Когда CRF получает `dima_bio` как фичу, он обучается: (а) доверять детектору на надёжных паттернах, (б) корректировать его на систематических ошибках. Это классический stacking — комбинация rule-based и statistical подходов.

### 6.4. Анализ ошибок CRF-10

43 из 50 тестовых предложений с ошибками (было 44 у CRF-3 — прогресс минимальный по количеству, но сами ошибки стали мельче).

**Типичные оставшиеся ошибки:**

**Splitting длинных спанов:**
- Gold: `Если для любого эпселен большего нуля, существует такое дельта больше нуля, то для всех точек M , принадлежащих области` → Pred: разбивает на 4 части (sent gosha_0001)
- Gold: `10 четвертых , 5 вторых плюс 5` → Pred: `10 четвертых` + `5 вторых` + `5` (sent gosha_0007)

**Over-extension:**
- Gold: `фиксирую` → Pred: `фиксирую у` (sent gosha_0002)
- Gold: `альфа` → Pred: `альфа коэффициент я` (sent gosha_0010)

**Пропуск коротких спанов:**
- Gold: `альфа-1, альфа-2 коэффициенты, альфа-2` → Pred: [] (sent gosha_0009)
- Gold: `Х не равняется 0` → Pred: [] (sent gosha_0018)

**Role analysis:** FN-токены по-прежнему доминируются ролью NONE (419 из 791) — модель пропускает токены, не входящие в словарь маркеров. FP аналогично (335 NONE из 668).

### 6.5. Топ фичи CRF-10

Топ-5 фичей для B-MATH/I-MATH:
1. `word[:3]:фик` (B-MATH): 2.96 — префикс «фик» (фиксировать, фиксированный)
2. `1:word.lower:,` (I-MATH): 2.44 — запятая справа = продолжение спана
3. `1:word.lower:положительным` (B-MATH): 2.29 — контекстное слово-триггер
4. `2:word.lower:действительным` (B-MATH): 2.28 — контекстный триггер на расстоянии 2
5. `1:word.lower:.` (B-MATH): 2.27 — точка справа = начало нового спана

Заметно, что `dep_label` и `dima_bio` не попали в топ-20 по абсолютному весу — их вклад распределён по множеству значений (30+ dep-ролей × 3 BIO-тега × позиции в окне). Эффект кумулятивный, а не сосредоточенный в одной фиче.

## 8. Ключевые выводы

1. **CRF-10 — лучшая модель проекта (span-F1 = 0.3612).** CRF с ручными фичами + dep features + stacking rule-based детектора превосходит fine-tuned ruBERT на +3.3 пп.

2. **Stacking rule-based и statistical подходов работает.** dima_bio как фича даёт +2.7 пп изолированно, +3.8 пп в комбинации с dep_label. Два разных «взгляда» на данные (rule-based + CRF) дополняют друг друга.

3. **POS избыточен при наличии dima_bio.** Детектор Димы уже использует POS внутренне, повторное добавление вредит.

4. **Три фактора успеха:** (а) качественные LLM-данные, (б) domain-specific фичи (math_role, is_math_word), (в) stacking rule-based детектора. Каждый даёт аддитивный вклад.

5. **Bottleneck остаётся прежним:** splitting длинных спанов при дискурсивных маркерах. Для дальнейшего роста нужны: данные с длинными спанами, содержащими «и», «,», «это»; или post-processing (склейка соседних предсказаний с gap ≤ 1).

## 9. Рекомендации для защиты

1. **Главный результат:** CRF с domain features (0.36) > ruBERT fine-tuned (0.33). На малых данных интерпретируемая модель с правильными фичами побеждает трансформер.

2. **Нарратив:** «Три ключевых компонента: (1) качественная LLM-генерация тренировочных данных, (2) domain-specific linguistic features, (3) stacking rule-based и statistical подходов. Каждый вносит измеримый вклад.»

3. **Интерпретируемость CRF** — преимущество для презентации. Можно показать топ-фичи, объяснить каждое решение модели. С BERT это невозможно.
